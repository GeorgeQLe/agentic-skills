Animation Audit — Pack → Drawer

A line-by-line reference for the /prototype card-pack opening animation in apps/skills-showcase.

Report only — this document changes no application code. It carries lightweight per-section feedback controls (👍 / 👎 / clarification) so you can flag any line that is wrong; there are no blocking approval gates and no compile-to-YAML.

File line numbers reflect the repository after the debug-harness work landed. The animation logic, motion values, and z-index behavior are unchanged — every transition is now wrapped in dbg.scaleT(...), which is an identity pass at speed 1×, and callback boundaries carry dbg.gate(...) / dbg.mark(...) that resolve synchronously when the harness is off. See the appendix.

Overview

A sealed pack (a foil-wrapped card box) sits on the grid. The user opens it one of three ways — tear the perforation, drag the top card up, or click the pack. A single preview card lifts out and, via a shared-layout layoutId morph, becomes card 0 inside a bottom-sheet drawer that fans the rest of the pack's cards out with a staggered spring. Dismissing the drawer reverses it: the fanned cards collapse back into card 0, the card morphs back into the pack, the sheet slides away.

The recurring bug. A one-frame card flash / z-index pop clusters at the apex — the moment the layoutId card morphs back from drawer into the pack during close. Two prior fixes (558a9873, fcc302a5) addressed it; it keeps returning because the animation is too fast to see what actually goes wrong. Section 5 dissects it.

Built on framer-motion 12.40.0: useMotionValue / useTransform, imperative animate() chains, shared-layout layoutId + LayoutGroup, spring physics, and AnimatePresence exits.

5 · Apex / z-index deep-dive the bug zone

The two guards that keep the card visible during the morph back

During close, the card must stay above the pack body while it morphs home, then drop behind it once it lands. Two pieces of state encode that.

// SealedPack.tsx:301  — render-phase z-index, opened branch
zIndex: (cardElevated || isClosingFromDrawer) ? 60 : undefined,

// SealedPack.tsx:241  — derived in render, not state
const isClosingFromDrawer = prevDrawerOpen.current && !isDrawerOpen;
  • cardElevated — state, set true both when lifting on open and when re-arming on the close layout-effect (:173); set false in the apex handler body (:334).
  • isClosingFromDrawer — a render-time derivation that is true for exactly the render where the drawer just closed (prevDrawerOpen still true, isDrawerOpen already false). It backstops cardElevated so the card never loses z-60 in the gap before the layout-effect runs.

The full z-index layer stack

Preview card (elevated)z-60
Sheet (drawer body)z-50
Scrim (backdrop)z-40
Shimmer foil overlayz-[2]
Pack body (bottom half / flap)z-[1]
Card at rest (no elevation)undefined → behind pack body
Why a flash is even possible. The card is z-60 (above everything) while elevated, but undefined (≈ behind the z-[1] pack body) at rest. The apex is the instant control passes from "morph finished, still z-60" to "elevation dropped, behind pack body". If the z-index drop and the final layout frame don't land on the same frame — or if the morph completes a frame before/after setCardElevated(false) commits — the card paints for one frame in the wrong stacking order: a visible pop/flash.

Timeline of the two prior fixes — and why it recurs

CommitChangeIntent
fcc302a5 Removed a packBodyElevated state flag; derived the card z-index in the render phase instead (the cardElevated || isClosingFromDrawer guard). Keep the card at z-60 through the layout animation from drawer back to apex, then let it drop behind the pack body on the descent — without a state round-trip that lagged a frame.
558a9873 In the close layout-effect, target the resting position (cardSlideY/cardDragY = 0) instead of the apex (y = −180); removed the now-redundant setTimeout + descent spring from onLayoutAnimationComplete, leaving a single layout motion. Let framer's layout animation carry the card smoothly drawer→pack in one motion, eliminating a second imperative spring that double-drove the card and flashed.
Why it keeps coming back. Both fixes are frame-timing fixes: they reduce the number of moving parts that can disagree on a frame, but the fundamental coupling remains — a render-phase z-index guard (isClosingFromDrawer, valid for one render) and a layout-animation-complete callback (setCardElevated(false)) must align with framer's final morph frame. Any change to spring/tween timing, an extra re-render, or a difference in when onLayoutAnimationComplete fires relative to the committed paint can re-open the one-frame window. The animation runs in ~300 ms, so the regression is invisible at full speed and each fix is validated by eye — which is exactly the gap the debug harness (appendix) closes.

Appendix · Debug harness (how to watch the apex)

An opt-in, inert-when-off harness lives in src/components/debug/ and is mounted on /prototype. With it disabled (default) the animation is unchanged. Enable it (the ⚙ gear, bottom-right) to:

  • Slow motion at 1× / 0.5× / 0.25× / 0.1× — implemented by scaling each authored transition (tween durations ×1/speed; springs via stiffness×speed², damping×speed, which preserves the damping ratio so overshoot is identical, just slower) plus the --debug-speed CSS variable for the keyframes.
  • Stepped mode — park the close chain at each boundary (especially layout-morph-out and drop-elevation) and advance one step at a time, holding the apex frame.
  • Live readout — cardElevated, the card's computed zIndex (60 vs unset), isClosingFromDrawer, isDrawerOpen, current step, current speed.
  • Drivers — Open (click) / Open (tear) / Close run the exact production callbacks on pack 0, so the real bug reproduces deterministically.

To catch the flash: enable → Open (tear) → Close in Stepped at 0.1×, step to sheet-exit → layout-morph-out, and pause before drop-elevation — the card is held mid-morph with cardElevated=true / z-60. Advance one step to watch the elevation drop and confirm whether the stacking order pops.
