Problem

Create Briefing Slides

The skill turns dense review surfaces into a visual deck that can be opened, navigated, commented on, and compiled into a structured handoff without replacing the underlying source artifacts.

Before

Long alignment pages, interrogation rounds, docs, and reports are accurate but slow to review.

During

The agent creates a deck with one primary idea per slide and inline review mechanics.

After

The reviewer sends structured YAML back to the owning workflow instead of scattered prose.

SKILL.md

Artifact Model

Deck first. References stay canonical.

The deck is the primary opened review surface. Dense pages and source documents remain linked references, not embedded content and not automatically opened.

Source And Output

`briefing-slides/*.html` is the visible output.

Path contract

Write decks directly under `briefing-slides/` and use `.html` so local `file://` review works.

Archive behavior

Before replacement, archive the previous deck under `docs/history/archive/YYYY-MM-DD/HHMMSS/briefing-slides/`.

Reference handling

Use relative reference links. Avoid `iframe`, `object`, `embed`, and automatic reference opens.

This gate is intentionally present so the dogfood deck exercises inline gate capture.

Convention

Presentation

It must behave like a review deck, not a markdown dump.

Navigation
  • Previous and next controls
  • Keyboard support for ArrowLeft, ArrowRight, Home, End, and Space
  • Hash resume links and filmstrip navigation
Status
  • Persistent slide counter
  • Progress track
  • One slide per page in print CSS
Skill workflow

Review Controls

Feedback belongs where the reviewer sees the issue.

Each slide can capture feedback type, annotations, and a status mark. Copy actions support slide titles, references, and compiled YAML with a textarea fallback when Clipboard API writes are unavailable.

Review controls

Handoff Contract

The compiler turns review state into routable YAML.

`briefing_slides`

Identifies the reviewed deck path.

`reference_pages`

Lists every linked dense page or source artifact.

`gate_answers`

Captures approval or revision gates.

`slide_feedback`

Records emphasize, revise, and clarification requests.

`annotations` and marks

Preserve reviewer notes and slide statuses.

`approval_status`

Stays `not-approved` until required gates and feedback are clean.

YAML contract

Dogfood Findings

The convention is concrete where implementation risk is highest.

Made clear
  • The deck is an additive review surface, not a replacement artifact.
  • The opener rule prevents accidental context switching into dense references.
  • The YAML contract gives reviewers a precise path back to the owning workflow.
May need tightening later
  • Reusable deck templates could reduce one-off JavaScript drift.
  • More exact approval rules may help distinguish optional feedback from blocking revision feedback.
  • Automated structural audits could enforce required controls faster than textual inspection.
