Alignment Page Example

This documentation-only example shows the dense review surface a producing AFPS skill can create before canonical writes or major handoffs. A real active page would live under alignment/ and follow the shared alignment-page convention.

What the page preserves

Source evidence

Links to research, design, prototype, spec, task, or code artifacts used by the skill.

Recommendation

Shows the proposed output or route with rationale and known tradeoffs.

Risks

Makes assumptions and unresolved uncertainty visible before approval.

Evidence matrix

This is what makes an alignment page a post-research surface: every claim is separated from the evidence that supports it, the inference drawn on top, and a confidence rating. The reviewer approves reasoning, not vibes — and can see exactly where a conclusion is thin.

Alternatives considered

A post-research surface shows the reasoning that was rejected, not just the recommendation — so approval is informed by what lost and why.

Rejected Guided multi-step wizard

Tested well for novices but added three screens before any value; conflicts with the first-success-in-one-session evidence. Lower confidence it beats a single connect step.

Deferred Sample-data preview before connect

Promising for reducing perceived risk, but no interview evidence that fake data builds trust. Parked pending a dedicated research round.

Rejected Persona-branched onboarding

Would serve power users, but the persona claim itself is low-confidence. Branching on an unverified assumption compounds risk.

Proposed file changes

Approval authorizes exactly these canonical writes — nothing is written until every gate below passes. The reviewer sees the blast radius before granting it.

Approval gates

Each gate is a real decision, not an acknowledgement. The compile banner stays blocked until every gate is approved — only then is the producing skill authorized to write. Revise or clarify holds the write.

Approval: blocked — 3 gates pending Approve all gates to authorize canonical writes
Scope gate

Target branch, artifact paths, and expected output are correct.

Evidence gate

Evidence is sufficient for the claims; low-confidence rows are acceptable as-is.

Artifact gate

The proposed file changes and downstream route are approved to write.

Compile responses

Approval is not a click — it is a payload. The page compiles every gate decision into a response YAML whose approval_status is computed from the gates' approval-effect metadata, not from label text. The user copies it, clears context, and pastes it into a fresh session; only a ready-for-agent-review payload authorizes the producing skill to write the canonical artifacts above.
