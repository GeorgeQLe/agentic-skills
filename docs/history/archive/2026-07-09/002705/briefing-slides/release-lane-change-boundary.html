Keep canary and general-release edits separate

A slide review surface for deciding how future skill, convention, package, and task changes declare their release lane before implementation.

Problem

The failure is lane mixing, not interrogation-specific

Observed risk

A general `master` edit can inherit canary-only conventions when the task plan mixes release lanes.

Broader blast radius
  • Skill frontmatter and conventions
  • Generator registries and package exports
  • Task docs and handoff language
  • Canary publish or promotion metadata
Boundary principles

Separate the lane before touching source

1Declare lane first

Every non-trivial task states its release lane before implementation.

2One source truth

Canonical skills stay in `packs/**` unless a promotion says otherwise.

3Promotion is explicit

Canary behavior graduates only through a named promotion task.

4Audit disallowed surfaces

General edits fail if canary-only surfaces appear.

Courses of action

Four practical options

A. Task metadata only

Fast and visible, but too soft without enforcement.

B. Lane contract + audit

Standardized fields plus mechanical checks. Recommended default.

C. Canary overlays

Useful for long-running experiments without duplicating skill trees.

D. Separate full source trees

Visible boundary, but high drift and hard promotion mechanics.

Directory split question

Do not split full skill directories by default

Why not
  • Duplicated mirror files
  • Duplicated archives and changelogs
  • Ambiguous package staging
  • Manual promotion diffing
Preferred boundary

Use one canonical source tree, release-lane metadata, branch policy, audit enforcement, and optional overlays for long-lived experiments.

Recommended standard

Release-lane contract + audit

Implementation shape

Minimal durable pieces

1. Convention doc

`docs/release-lane-convention.md` defines lanes, branch rules, allowed and disallowed surfaces.

2. Task block

`tasks/todo.md` includes a required `### Release Lane` block for non-trivial work.

3. Audit script

`scripts/audit-release-lane.mjs` validates active lane, branch, and touched surfaces.
