# Plan: Create `/uat-guide` Skill

## Context

The `/uat` skill generates UAT journeys with high-level task sequences ("sign up", "navigate to dashboard"), but a tester unfamiliar with the product needs click-by-click guidance through those steps. The `/guide` skill provides that click-by-click pattern but is oriented around infrastructure blockers (OAuth, DNS), not UAT journeys. A new `/uat-guide` skill bridges the gap: it reads the UAT plan, picks a journey, and expands each task-sequence step into detailed tester instructions.

**Why a separate skill (not extending `/uat`):** UAT is planning/analysis (generates journeys), uat-guide is interactive walkthrough (consumes journeys). Different modes, different shipping contracts, different invocation patterns. UAT already routes to `/guide` for blockers — `/uat-guide` fits naturally into that routing chain.

## Files to Create

### 1. `global/claude/uat-guide/SKILL.md`

New skill file with:

**Frontmatter:**
- name: `uat-guide`
- description: "Expand a UAT journey into click-by-click tester instructions, then update the result log on completion"
- type: `analysis`
- version: `1.0.0`
- argument-hint: `"[optional: journey number, journey name, or 'next']"`

**Workflow (8 steps):**

1. **Locate UAT plan** — Read `research/uat-plan.md`. If missing, stop and recommend `/uat`.
2. **Select journey** — Parse `### Journey N:` headings + result log status. If args provided, match by number or name. Otherwise pick first `Status: Not run`. If all done, tell user and offer re-run.
3. **Gather project context** — Read `.agents/project.json`, README, CLAUDE.md, specs, docs, tasks. Search codebase for routes, UI components, CLI entry points, API endpoints, env vars, config, navigation structure.
4. **Detect product interface type** — Web app → click-by-click, CLI → command-by-command, API → request-by-request, Hybrid → mixed.
5. **Research current instructions** — Web search for external services/platforms referenced in the journey. Same rationale as `/guide`.
6. **Expand task sequence** — For each step: numbered sub-steps with exact UI elements/commands/requests, checkpoints tied to acceptance criteria, evidence capture points, gotchas. Include preparation section from Setup field and final verification from acceptance criteria.
7. **Present guide** — Output directly in conversation (not to file).
8. **Collect results** — After user reports completion: update `#### UAT result log` inline in `research/uat-plan.md`, check off item in `tasks/manual-todo.md`. If Fail/Blocked, suggest follow-up routing.

**Constraints:**
- Always web search for external UIs
- Project-specific values, no generic placeholders
- Read-only except result log + manual-todo checkbox
- No shipping contract
- One journey at a time
- Don't execute the product
- Don't mark complete unprompted
- Don't invent acceptance criteria
- Handle all product surface types
- Prerequisite: UAT plan must exist

**Output format template** — Same structure as guide but journey-oriented: preparation section, numbered steps with checkpoints/evidence/gotchas, final verification checklist, non-acceptance signals, tester notes prompt.
