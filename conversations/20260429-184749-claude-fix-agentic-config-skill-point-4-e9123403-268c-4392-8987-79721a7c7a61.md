## What Went Well

1. **Spec phase was solid.** The sequential spec-interview process (04/24–04/26) across 9 game variants completed cleanly — 10+ Claude sessions produced specs and plans without conflicts.

2. **Per-package `todo.md` setup was correct.** Giving each package its own plan file with independent phase numbering was the right structural call for enabling parallelism.

4. **Wave 2+ showed reduced pnpm-lock contention.** The first wave had sessions with 85–121 pnpm-lock references each. Later waves dropped to 2–10 per session, suggesting some learning/adjustment was happening.

## What Went Wrong

### Root Cause: Shared `pnpm-lock.yaml` Contention

The monorepo has a single `pnpm-lock.yaml` at the root. All 9 Codex agents ran `pnpm install` or `pnpm add` to install package-specific dependencies (pixi.js, three.js, phaser, etc.), and each run **rewrites the entire lockfile**. With 9 agents running concurrently:

- Agent A writes `pnpm-lock.yaml` for submarine-combat deps
- Agent B writes `pnpm-lock.yaml` for auto-battler deps, **overwriting A's changes**
- Agent A detects the lockfile changed and retries
- Infinite loop of lockfile thrashing

The evidence is clear: the submarine-combat first-wave session alone had **121 pnpm-lock references** and **101 distinct pnpm-lock related events** — mostly reading, diffing, and re-running installs as the file kept changing under its feet.

### Secondary Issues

1. **No isolation between agents.** All 9 Codex agents operated on the **same git worktree**. Beyond pnpm-lock, they were also seeing each other's uncommitted changes via `git status` (the session logs show `git status` returning modified files from **other packages** — auto-battler seeing colony-sim changes, etc.).

2. **Agent churn was expensive.** 194 sessions × ~180 response items average = massive compute spend. Most of it was wasted on retry loops and re-reading the same files.

3. **No coordination layer.** Each agent independently decided to run `pnpm install` as its first step. There was no mechanism to run dependencies once and then dispatch pure-code-generation tasks.

4. **Codex lacks worktree isolation.** Unlike Claude Code's `isolation: "worktree"` parameter, Codex sessions all share the working directory, making true parallel monorepo work impossible for tasks that touch shared files.

## Recommendations for Future Parallel Work

### 1. Pre-install all dependencies before parallelizing
Run a single session that adds all packages' dependencies to `package.json` files and runs `pnpm install` once. Then dispatch parallel agents that only write source code — never touching `pnpm-lock.yaml` or running `pnpm install`.

### 2. Use git worktrees for isolation
Create one git worktree per package (`git worktree add ../bismarck-v5-submarine packages/submarine-combat`). Each Codex agent works in its own worktree. Merge results back sequentially.

### 3. Serialize lockfile-touching work, parallelize code-only work
Split each package build into two phases:
- **Phase A (serial):** Add deps to all `package.json` files, run `pnpm install` once, commit the lockfile
- **Phase B (parallel):** Each agent writes only source files within its own `packages/X/src/` directory

### 4. Add a CLAUDE.md/AGENTS.md rule for monorepos
Add this to the project config:
```
NEVER run `pnpm install`, `pnpm add`, or modify `pnpm-lock.yaml` when running as a parallel agent. All dependency changes must be pre-staged before parallel work begins.
```

### 5. Consider Claude Code subagents instead
Claude Code's `Agent` tool with `isolation: "worktree"` provides per-agent git isolation that Codex currently lacks. For monorepo parallel work, Claude Code subagents may be more reliable than multiple Codex instances.

### 6. Batch parallel sessions with a "no shared file" guard
Before launching N agents, validate that their planned work scopes don't overlap on any shared files. If they do, resolve the shared-file changes first.
