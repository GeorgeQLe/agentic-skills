Pack Skill Sunset Plan

Alignment date: 2026-06-16 | Category: Utility & Maintenance | Visual tier: document | Artifact: alignment/pack-skill-sunset-plan.html

This page is a decision review for sunsetting the active pack skill. It records current dependency surfaces, a conservative replacement model, readiness blockers, phased implementation, and gates that must be approved before any implementation begins.

Pre-approval stop: no skill, launcher, CLI, documentation, or test surface should be removed, deprecated, or rewritten from this page alone. Implementation starts only after bottom compiled response YAML has approval_status: ready-for-agent-review.

Recommendation Summary

Default recommendation

Do not fully sunset pack yet. Keep the active skill and shell launcher until Node CLI ownership covers the remaining shell-backed surfaces and package verification is green.

Best near-term target

Deprecate user guidance for /pack and $pack only after npx skillpacks / gskp owns the same workflows without relying on shell fallbacks.

Conservative end state

Retain a thin compatibility shim for one release cycle after docs and root instructions route users to the npm CLI.

Hard stop

Do not remove the skill while root instructions still use scripts/pack.sh which as the missing-skill fallback.

Sunset Definition

"Fully sunset" can mean three different outcomes. The review decision should pick one target before implementation begins.

Option Meaning Advantages Risks Recommendation
Remove pack skill Delete active base/claude/pack and base/codex/pack skill roots after docs and installed mirrors are updated. Eliminates duplicate agent entry points and forces a single CLI route. Breaks users and agents trained to use /pack or $pack; risky while root instructions still mention shell fallback. Not recommended as the first implementation step.
Deprecate and retain compatibility shim Update skill text to say it is deprecated, route explicit work to npx skillpacks / gskp, and retain minimal fallback guidance for one release cycle. Preserves existing workflows while shifting users to the replacement route. Still carries a visible old entry point temporarily. Recommended after CLI parity is complete and verified.
Keep as active facade Keep pack as the agent-facing UX and let it call npm or shell helpers internally. Lowest disruption for agents. Does not accomplish a true sunset; keeps skill registry clutter. Acceptable only if the goal changes from sunset to stabilization.

Target Replacement Model

The target model should make the npm CLI the canonical route and reduce the skill to either no surface or a temporary compatibility notice.

Canonical user command

npx skillpacks <command> and the gskp alias own pack lifecycle behavior for package consumers.

Source checkout fallback

scripts/pack.sh remains documented only for maintainers or explicitly shell-backed commands until parity is complete.

Agent missing-skill lookup

Root instructions should use npx skillpacks which <skill> or a Node-owned lookup before they stop mentioning scripts/pack.sh which.

Guided setup

Replace skill-mediated guided setup with either a Node-owned recommend flow or explicit docs that require user confirmation before installing packs.

Readiness Blockers

Blocker Why it blocks sunset Required proof Owner surface
Node-owned recommend The guided skill flow currently starts from scripts/pack.sh recommend. Removing the skill without a replacement loses the recommendation checkpoint. npx skillpacks recommend works without shell fallback and has tests for repo-signal classification. packages/skillpacks/src/cli/*, package tests, docs.
Node-owned which Root instructions use scripts/pack.sh which for missing-skill fallback. This is the most direct removal blocker. npx skillpacks which <skill> resolves pack, install state, and hibernated pack cases without shell fallback. run-pack-script.mjs, manifest lookup helpers, root instructions.
install-deck ownership decision Docs and tests state deck materialization still requires bash and jq through packaged shell backend. Either port deck materialization to Node or explicitly classify it as a permanent shell compatibility surface. docs/decks.md, compatibility matrix, CLI tests.
Root instruction migration AGENTS.md and CLAUDE.md currently name the shell lookup fallback. Root docs route to npm CLI lookup and preserve reload guidance for Claude and Codex. Provisioned instruction source plus generated root files.
Docs and tests cleanup Public docs still teach /pack, $pack, and source checkout helper usage in places. Docs consistently distinguish canonical npm route from maintainer-only shell fallback; compatibility tests match the new contract. README.md, docs/*, packages/skillpacks/test/*.
Package test health Sunset changes affect install, refresh, docs, compatibility, and active skill registry behavior. Package node tests, build check, alignment audit, skill mirror/version checks, and relevant layer1 routing tests pass. Package test suite and repo-level verification scripts.

Implementation Phases

Phase Work Exit criteria Proceed gate
1. Assessment Inventory active skill, shell, npm, docs, tests, installed roots, generated package outputs, and root instruction references. Evidence table is current and all shell-backed commands are classified. Approve current dependency map.
2. Node parity Port or explicitly classify recommend, which, and install-deck. Add tests before changing docs. CLI ownership matrix is green and no accidental shell fallback remains for commands claimed as Node-owned. Approve CLI ownership decision.
3. Agent instruction migration Update root instruction source and generated root docs so missing-skill fallback uses the canonical CLI route. Generated AGENTS.md and CLAUDE.md no longer require scripts/pack.sh which unless that is deliberately retained. Approve user-facing route.
4. Docs and tests cleanup Update README, skills reference, Codex workflow, decks docs, compatibility matrix, and regression tests. Docs consistently separate package consumer, source maintainer, and compatibility routes. Approve docs/test contract.
5. Deprecation or removal Either rewrite pack as a deprecation shim or remove it from active base mirrors with archives/changelogs as required. Skill versioning and archive rules are satisfied; installed roots/package builds are refreshed if behavior changes. Approve selected sunset target.
6. Release verification Run package, skill, docs, alignment, and install-route verification; update ship manifest; commit and push. All required checks pass or any pre-existing failure is proven against unmodified baseline. Approve release readiness.

Release Verification

The final implementation should use a wider verification set than this planning page because it would touch active skill contracts, generated package outputs, public docs, and CLI behavior.

Page-only checks

node scripts/audit-alignment-pages.mjs, TTS injection if missing, and git diff --check.

CLI checks

npm --workspace packages/skillpacks run test:node and npm --workspace packages/skillpacks run build:check.

Skill contract checks

bash scripts/skill-archive-audit.sh --strict, bash scripts/skill-versions.sh --missing, and mirror parity checks after skill edits.

Routing checks

Focused layer1 tests for install routing, skill next-step routing, and root instruction generated-block integrity.
