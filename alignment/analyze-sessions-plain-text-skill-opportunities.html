Method

Section feedback:

I parsed the full available compact history files at ~/.claude/history.jsonl and ~/.codex/history.jsonl. For Codex, I also scanned ~/.codex/sessions/**/*.jsonl for session metadata such as cwd, model, provider, and session id. I used the compact Codex history as the prompt-count source to avoid double counting rich rollout records.

Plain-text records exclude prompts that start with / or $. The analysis then combined exact normalized repeats, keyword clusters, project/source breakdowns, and manual review of representative examples. Counts labeled as "classifier" are evidence for magnitude, not exact semantic truth, because keywords such as "current" and "overlap" can be noisy.

Script Extraction Clarification (2026-06-07)

Section feedback:

Follow-up finding: not every recurring natural-language workflow should become a new skill. Deterministic, repeatable primitives should become scripts or script-backed utilities that skills can call, while skills should retain policy, judgment, sequencing, and user-facing synthesis.

This clarification narrows the original candidate list. Repeated plain-text asks are still useful evidence, but the implementation choice should split "agentic workflow" from "deterministic primitive." scripts/pack.sh is the strongest current example of this split: the script owns the repeatable lifecycle, while the skill layer handles routing, explanation, and reload guidance.

The primary extraction candidate from the scriptability follow-up is compile-central-alignment / alignment index maintenance. The current contract fully specifies a deterministic scan, grouping, sorting, and HTML generation algorithm for alignment/index.html, but only the browser opener is delegated to a script. That index-generation algorithm is a better script-backed utility candidate than a brand-new natural-language skill.

Extraction should also inherit the portability lessons from downstream inventory: manual utilities need to work under macOS stock Bash 3.2 or explicitly declare Bash 4+ or Node. The same standard should prevent path-level confusion such as recommending scripts/init-agentic-skills.sh when the actual launchers live elsewhere.

Alternatives Considered

Section feedback:
  • Do nothing: rejected for high-frequency patterns because the same phrases keep recurring across projects and sources.
  • Make every pattern a new skill: rejected because several should be aliases, subroutines, or existing skill enhancements.
  • Route all plain text to exec: too coarse for deploys, review remediation, and instruction updates where pack-specific gates matter.
  • Use only standing instructions: useful for freshness and task-doc habits, but insufficient for staged deploys, skill visibility repair, and review fixes.
